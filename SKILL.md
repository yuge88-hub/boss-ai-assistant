# BOSS直聘求职助手

六阶段全自动求职流水线：发现 → 筛选 → 诊断 → 执行 → 监控 → 报告。

## 安装（首次使用，只需一次）

```bash
# 1. 确保已安装 opencli
npm i -g @jackwener/opencli

# 2. 进入 skill 目录，运行安装脚本
cd ~/.claude/skills/boss-ai-assistant
bash scripts/install.sh
```

安装脚本做了两件事：
1. 把 `search-greet.js` 和修复版 `send.js` 复制到 opencli 全局目录
2. 注册 `search-greet` 命令

安装后 `opencli boss search-greet` 即可用。

## 使用前确认

- 简历已上传到 BOSS 平台（手动上传或通过 `opencli boss`）
- 手机号已知（需要时在回复中给出）
- 已在 Chrome 登录 [zhipin.com](https://www.zhipin.com)

## 全局规则

### 页面管理
每个 `opencli boss` 命令可能第一次报 `stale page identity`，立即重试一次即可。如果连续两次 stale，杀守护进程：

```bash
powershell -Command "Get-Process node | Where-Object { $_.CommandLine -like '*daemon*' } | ForEach-Object { Stop-Process -Id $_.Id -Force }"
sleep 5
```

### 环境变量
批量搜索打招呼必须设置超时：
```bash
OPENCLI_BROWSER_COMMAND_TIMEOUT=300
```

### 验证标准
- 发简历/换微信/换电话：看 `--verbose` 输出中 `sureFound: true`
- 打招呼：返回 `changed-to-continue`
- 发消息：返回 `status: sent`
- **不只看 CLI 返回的 status，必须看 verbose debug 输出确认**

## Phase 0：前置检查（仅回头客）

**如果是第一次使用本 skill，直接跳到 Phase 1。**

只有之前运行过且项目中有 `memory/boss-progress.md` 文件时才执行此阶段：

1. 检查 `memory/boss-progress.md` 是否存在（项目根目录下）
2. 如果存在 → 读取上次 A/B/C 分类和联系方式
3. 如果不存在 → 跳过，Phase 5 结束时会自动创建
4. 运行 `opencli boss chatlist --limit 50 -f json` 获取当前聊天列表
5. 合并去重：已在 progress.md 中标记为"已完成全部操作"的跳过

## Phase 1：发现新机会

**目标**：搜索指定关键词，批量打招呼。

**关键词由用户配置**，如用户未指定，询问用户想搜索什么岗位。示例：

```bash
# 用户指定关键词后执行（city 和 limit 可调整）
OPENCLI_BROWSER_COMMAND_TIMEOUT=300 opencli boss search-greet "用户的关键词" --city 北京 --limit 15 --verbose
```

每个搜索完成后记录：搜索词、打招呼数量、验证状态。

---

## Phase 2：筛选对话

**目标**：拉取聊天列表，按规则筛选出需要处理的。

```bash
opencli boss chatlist --limit 50 -f json
```

**过滤规则**（首次使用时询问用户想要关注和忽略的岗位类型，后续记住偏好）：
- 关注（保留）：用户指定的目标岗位关键词，如 `营养|健康|医|药|销售|顾问`
- 忽略（跳过）：用户指定不相关的岗位关键词，如 `司机|网约车|驾驶|家政|建筑`

**分类标记**：
- 新对话：首次出现，无任何操作记录
- 有回复：对方发了新消息
- 已处理：按钮全部 unable + 消息已回复

---

## Phase 3：智能诊断

**目标**：对每个保留的对话，判断当前状态并决定下一步动作。

### Step 1：看按钮状态

进入对话后，从 `--verbose` debug 输出检查按钮：
- `发简历 btn: unable` → 简历已发，跳过
- `换微信 btn: unable` → 微信已换，跳过
- `换电话 btn: unable` → 电话已换，跳过

### Step 2：看最新消息，分类

| 类型 | 特征 | 动作 |
|------|------|------|
| 🔴 面试邀请 | 含"面试"+"时间"/"邀请"/"确认" | 回复确认参加 → **A类** |
| 🟠 要手机号 | "电话"/"手机号"/"语音说号码" | 回复文字直接给出手机号 |
| 🟠 要微信 | "微信"/"加一下"/"联系方式" | 回复 + `--exchange wechat` |
| 🟡 真人提问 | 手动打的、有具体问题 | 针对性回复 + 没发过就发简历 |
| 🟡 自动回复 | "我有一定的XX经验，不甘于…" | 回复 + 发简历 |
| 🟡 平台警告 | "保障您的安全" | 回复（表示理解）+ 发简历 |
| 🟢 对方已同意 | 系统消息"对方已同意" | 不回复 → **B类** |
| 🟢 语音消息 | `[语音]` 标识 | 回复"语音暂不便收听，请发文字" |
| 🟢 图片消息 | `[图片]` 标识 | 回复"图片暂不便查看，请文字说明" |
| ⚪ 空白对话 | 无对方消息 | 发开场消息 + 发简历 |
| ⚪ 简短回应 | 表情包/"好的"/"嗯" | 继续回复维持热度 |

### Step 3：判断对话深度，定分类

| 分类 | 标准 |
|------|------|
| **A类** | 消息明确含面试邀请或已确认面试时间 |
| **B类** | 真人回复 ≥ 2轮 + 联系方式已交换 + 简历已发 |
| **C类** | 刚打招呼 / 自动回复 / 只有1轮互动 |

### Step 4：生成回复内容

**核心原则：不千篇一律**。根据岗位名和对方消息自然生成。

```
面试邀请："感谢邀请！确认参加面试，请问具体时间和地点？"

要手机号："好的，我手机号是 1XXXXXXXXXX，(名字)，随时联系。"

要微信："好的，已通过平台发送微信交换请求，您通过一下。"

真人提问：根据提问内容针对性回答

自动回复："您好，我对贵公司(岗位名)很感兴趣，有相关经验，方便详细聊聊吗？"

平台警告："您好，感谢提醒。我对(岗位名)很感兴趣，方便在平台内沟通一下吗？"

语音/图片："抱歉，语音/图片暂不方便，方便发文字说明吗？"

开场消息："您好，看到您的(岗位名)招聘，我对这个岗位很感兴趣，有相关经验，方便详细聊聊吗？"
```

### Step 5：决定执行动作

按优先级排序：
1. 面试邀请 → 只回复确认，不干别的
2. 对方要手机号 → 回复里直接给号码（不通过平台交换）
3. 对方要微信 → 回复 + `--exchange wechat`
4. 真人提问/自动回复/空白 → 回复 + 没发过简历就发简历
5. 已有联系方式 → 记录到 progress.md

---

## Phase 4：执行操作

**执行顺序**：先发消息 → 再发简历 → 最后换联系方式

```bash
# 发消息
opencli boss send "<人名>" "消息内容" --side geek --verbose

# 发简历
opencli boss send "<人名>" --exchange resume --side geek --verbose

# 换微信
opencli boss send "<人名>" --exchange wechat --side geek --verbose

# 换电话
opencli boss send "<人名>" --exchange phone --side geek --verbose
```

**重要**：
- 用人名匹配，不用公司名（公司名会被截断）
- 每个操作检查 verbose 输出确认 sureFound: true 或 status: sent
- 按钮已是 unable 的跳过
- 同一守护进程内尽量连续执行完所有操作

---

## Phase 4.5：循环监控

执行完一轮后，等 5~10 分钟，重新拉 chatlist 看有无新回复。

```
循环：
  等 5~10 分钟
  拉 chatlist → Phase 2 筛选 → Phase 3 诊断 → Phase 4 执行
  停止条件（满足任一）：
    1. 连续 2 轮无新回复
    2. 检测到 A 类面试邀请 → 立刻停止出报告
    3. 用户手动叫停
    4. 已达最大轮数 3 轮
```

## Phase 5：输出报告

### 报告格式

```markdown
# BOSS直聘求职报告 (日期)

## A类 - 已约面试 (N人)
| 姓名 | 公司 | 岗位 | 薪资 | 面试时间 | 联系方式 |
|------|------|------|------|----------|----------|

## B类 - 深度沟通 (N人)
| 姓名 | 公司 | 岗位 | 轮数 | 联系方式 | 缺什么 | 建议 |
|------|------|------|------|----------|--------|------|

## C类 - 初步接触 (N人)
| 姓名 | 公司 | 岗位 | 状态 | 建议下一步 |
|------|------|------|------|-----------|

## 联系方式汇总
| 姓名 | 微信 | 电话 | 来源 |
|------|------|------|------|

## 今日操作统计
- 搜索打招呼：N个
- 发简历：N份
- 换微信：N次
- 换电话：N次
- 回复消息：N条

## 建议
- 需立即处理：面试邀请等
- 需跟进：B类中缺动作的
- 可忽略：已标记的
```

### 更新记忆

将报告摘要写入项目根目录下的 `memory/boss-progress.md`，格式：
- A/B/C 分类表格
- 联系方式汇总
- 今日操作统计

这样下次使用时 Phase 0 可以读取继续跟进。

## 常见问题

**Q: 页面一直 stale page identity？**
A: 杀守护进程 + 等5秒 + 重试。最多重试3次。

**Q: 换微信 confirmed 但对方没收到？**
A: 看 verbose 输出中 sureTxt 是否为"确定"。如果按钮已是 unable 说明之前已成功。

**Q: 搜索打招呼找不到职位卡片？**
A: 第一次可能 stale page，重试一次。确认搜索URL是 `/web/geek/jobs`（复数）。

**Q: 多个同名联系人（如多个"郭女士"）怎么办？**
A: 用公司名辅助匹配：`opencli boss send "步步高升" ...`

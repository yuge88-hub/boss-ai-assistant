> 2026年5月27日，用一个下午的时间，完成了从"按钮点不中"到"12个营养健康类对话全自动回复"的迭代过程。本文是一份实战复盘，记录每一步遇到的问题、分析过程和解决方案。

## 一、起点：一个按钮引发的血案

下午5点，我开始对A类客户（已约面试的7家公司）执行批量操作：发简历、换微信、换电话。命令很简单：

```bash
opencli boss send "孙女士" --exchange resume --side geek
```

返回了 `status: exchanged`，看起来成功了。

但用户看了一眼BOSS页面，说：**"你根本没操作成功。"**

这就暴露了第一个问题：**CLI返回的成功不等于实际操作成功。**

### 问题分析

打开 `--verbose` 调试输出，看到了关键信息：

```
Exchange debug: {
  "clickCls": "toolbar-btn-content",  ← 点了什么？
  "sureFound": false,                  ← 确认弹窗没找到！
  ...
}
```

`toolbar-btn-content` 是什么？查看页面DOM结构后发现，BOSS聊天界面的底部工具栏中，每个按钮由两个元素组成：

| 元素 | Class | 作用 |
|------|-------|------|
| 文字容器 | `toolbar-btn-content` | 纯展示文字，无事件绑定 |
| 真正按钮 | `toolbar-btn tooltip tooltip-top` | Vue点击事件绑在这里 |

代码遍历所有 `[class*="toolbar"]` 元素，找到第一个文本匹配的就点击——总是先命中 `toolbar-btn-content`，点了个寂寞。

### 修复

```javascript
// 跳过仅有 toolbar-btn-content 不含 toolbar-btn 的文字容器
if (btn.classList.contains('toolbar-btn-content')
    && !btn.classList.contains('toolbar-btn')) {
  continue;  // 跳过，找真正的按钮
}
```

修复后 `clickCls` 变成 `toolbar-btn tooltip tooltip-top`，正确的按钮被点击了。

## 二、弹窗确认按钮消失了？

按钮点击正确了，但 `sureFound: false`——弹窗的确定按钮找不到。

代码原逻辑：点击工具栏按钮 → 等800ms → 找 `.btn-sure-v2` → 点击确认。

两个问题：
1. **800ms不够**，弹窗动画还没渲染完
2. **换微信的弹窗结构不同**——它不是 `.btn-sure-v2`，而是一个 `sentence-popover panel-wechat` 面板

### 修复

```javascript
// 1. 延长等待到1500ms
// 2. 多级搜索确认按钮
const sureSelectors = [
  '.btn-sure-v2', '.btn-sure', '.btn-confirm',
  '[class*="sure"]', '[class*="confirm"]',
  '.van-dialog__confirm', '.btn-primary', '.btn-ok'
];
// 3. 兜底：按文字搜索"确定"/"确认"/"发送"
// 4. 只搜真正的按钮元素，避免点到弹窗面板本身
```

修复后，三个操作的确认按钮全部正确命中：

| 操作 | sureCls | sureTxt | sureFound |
|------|---------|---------|-----------|
| 发简历 | `btn-v2 btn-sure-v2` | 确定 | ✅ |
| 换电话 | `btn-v2 btn-sure-v2` | 确定 | ✅ |
| 换微信 | `btn-v2 btn-sure-v2` | 确定 | ✅ |

## 三、幽灵般的页面过期

搞定了按钮逻辑，又遇到了新问题：**每次只能执行一次操作，第二次必定报错。**

```
Page not found: XXX — stale page identity
```

`opencli boss` 的每个子命令使用独立的浏览器页面池。执行一次 `send` 后页面被释放，再次调用时分配到过期页面。每次出现的Page ID都不同，说明整个页面池都过期了。

### 解决方案

摸索出一个固定流程：

```bash
# Step 1: 杀守护进程（Windows下中文路径只能用PowerShell杀进程）
powershell -Command "Get-Process node | Where-Object { $_.CommandLine -like '*daemon*' } | ForEach-Object { Stop-Process -Id $_.Id -Force }"

# Step 2: 等守护进程自动重启
sleep 5

# Step 3: 预热页面池（第一次可能stale，第二次可用）
opencli boss chatlist --limit 2 -f json

# Step 4: 执行真正的操作
opencli boss send "人名" --exchange resume --side geek --verbose
```

**关键发现：不要用 `--keep-tab true`**，它会持有过期页面导致持续失败。

稳定后，同一守护进程内可以连续执行多个操作（已验证连续7个操作不掉线）。

## 五、新建命令：搜索即沟通

之前的流程依赖已有的聊天列表。但对于新职位，需要：搜索 → 点击职位 → 点击"立即沟通"。

这个功能CLI还没有，于是新建了 `search-greet.js`。

### 技术要点

1. **页面结构探索**：先dump页面DOM找到正确的选择器
   - 职位列表：`.rec-job-list > li`
   - 职位名：`[class*="job-name"]`
   - 搜索URL是 `/web/geek/jobs`（注意复数）
2. **点击后验证**：点完"立即沟通"后等1.5秒，检查按钮是否变成"继续沟通"
3. **超时处理**：15个职位需要约3分钟，必须设 `OPENCLI_BROWSER_COMMAND_TIMEOUT=300`

### 使用方式

```bash
OPENCLI_BROWSER_COMMAND_TIMEOUT=300 \
  opencli boss search-greet "营养师" --city 北京 --limit 15 --verbose
```

### 验证标准

| 返回值 | 含义 |
|--------|------|
| `changed-to-continue` | 按钮确实变了，发送成功 |
| `already-greeted` | 之前已打过招呼 |
| `btn-still-there` | 点了但按钮没变，可能失败 |
| `no-greet-btn` | 根本没有"立即沟通"按钮 |

## 七、核心踩坑总结

### 1. 按钮选择的陷阱
BOSS的Vue组件中，按钮由文字容器+可点击元素组成。**永远点包含 `toolbar-btn` 类的元素，跳过仅有 `toolbar-btn-content` 的元素。**

### 2. 验证不能偷懒
CLI返回 `status: exchanged` 不代表成功。必须检查：
- `sureFound: true`（确认弹窗找到了）
- `clickCls` 包含 `toolbar-btn`（点的是按钮不是文字）
- 对于打招呼：按钮必须变成"继续沟通"

### 3. 页面过期是常态
每次守护进程重启后，第一个命令大概率失败，重试即可。稳定的流程是：杀进程 → 等5秒 → 预热 → 执行。

### 4. 命名匹配的坑
- 聊天列表用**人名**匹配（公司名会被截断）
- 搜索页面用**职位名**匹配（`.rec-job-list > li` 中的 `[class*="job-name"]`）
- 处理多个人时要区分重名（如多个"郭女士"，要用公司名辅助定位）

### 5. 环境变量的重要性
`OPENCLI_BROWSER_COMMAND_TIMEOUT=300` 对于批量操作必不可少，默认60秒只能处理约6个职位。
